## Storage Systems
To support the four goals of the hospital AI system, a combination of storage systems is used, each optimized for a specific workload.
For predicting patient readmission risk, a data lakehouse (e.g., Delta Lake / Parquet on object storage) is used. Historical treatment data is stored in structured formats, enabling efficient batch processing and model training. This allows scalable machine learning pipelines and supports large volumes of clinical data.
For natural language querying of patient history, a hybrid system combining OLTP database + vector database is used. The OLTP database (e.g., PostgreSQL) stores structured patient records, while a vector database (e.g., FAISS or Pinecone) stores embeddings of patient notes and reports. This enables semantic search and supports LLM-based querying in plain English.
For monthly reporting, a data warehouse (e.g., Snowflake/BigQuery-like architecture) is used. Aggregated and cleaned data from the lakehouse is loaded into a warehouse optimized for analytical queries. This ensures fast performance for dashboards like bed occupancy and department-wise costs.
For real-time ICU vitals streaming, a time-series database (e.g., InfluxDB) or streaming system (Kafka + storage) is used. These systems are optimized for high-throughput, low-latency ingestion of continuous data streams and allow real-time monitoring and alerting.
## OLTP vs OLAP Boundary
The OLTP system consists of operational databases such as PostgreSQL, which handle real-time transactional data like patient records, doctor inputs, and ICU device ingestion. These systems prioritize low latency and high consistency.
The OLAP system begins once data is moved into the data lakehouse and data warehouse layers. ETL/ELT pipelines transform raw data into structured analytical formats. The lakehouse stores raw and semi-processed data, while the warehouse stores curated, aggregated datasets for reporting and analytics.
Thus, the boundary lies at the data ingestion and transformation layer, where operational data is replicated or streamed into analytical storage systems. OLTP supports real-time operations, while OLAP supports analytics, ML, and reporting.
## Trade-offs
One significant trade-off in this architecture is increased system complexity due to the use of multiple storage systems (OLTP, lakehouse, warehouse, vector DB, and streaming systems). Managing data consistency, pipelines, and integration across these systems can be challenging.
To mitigate this, a unified data orchestration and governance layer (e.g., Airflow + data catalog) can be implemented. This ensures proper data lineage, scheduling, and monitoring of pipelines. Additionally, adopting a lakehouse architecture as the central data layer reduces duplication and simplifies data access for both analytics and ML workloads.
